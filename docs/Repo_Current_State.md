The repo contains a Vite + React + TypeScript app with a simple editor shell. The upload area accepts one PNG, JPG, JPEG, or WEBP image and stores source image metadata in app state.

The current UI uses a modest comfort spacing pass across the shell, header actions, workspace grid, panels, upload area, inspector forms, timeline rows, editor canvas, and preview area. This is CSS-only layout polish and does not change workflow or project data behavior.

After an image is selected, the app displays:

- File name
- Source image width
- Source image height
- MIME type, when available

The editor canvas now displays the uploaded page image using the stored object URL. The page is centered, constrained to the available editor area, and rendered without distortion.

After an image is uploaded, the app creates one default camera shot in source image coordinate space. The shot is centered inside the image bounds and rendered as a selectable overlay on top of the displayed page.

Camera shots are flexible subject regions. They can be tall, wide, square, or otherwise non-16:9 while staying inside the source image bounds.

The preview player remains a fixed 16:9 video stage. Camera shot boxes define the subject area that must be fully visible inside that stage. Non-16:9 camera shots are rendered inside a centered clipped shot window with contain-style scaling, so empty stage space is neutral preview background instead of surrounding page content.

Focus regions are stored separately from camera shots as page-level annotations. They use source image coordinate space, support free-ratio dimensions, include a `kind` such as panel, speech, face, detail, action, or other, include an optional `effectType` such as lift, spotlight, zoom, or none, and include an optional numeric `sequenceOrder` for deterministic attention ordering. A focus region may keep optional `sourceShotId` metadata for selection context, but its existence does not depend on that shot.

Camera shots and focus regions now support optional purpose metadata. `shotPurpose` and `focusPurpose` use the shared purpose values `establishing`, `panel`, `dialogue`, `reaction`, `emotion`, `action`, `detail`, `reveal`, `transition`, and `other`. Purpose describes reading/storytelling intent, not visual treatment. It is persisted in Project JSON when present and ignored when absent. Manual purpose controls are exposed in the inspector for selected camera shots and selected focus regions. Changing purpose metadata alone does not affect preview playback, canvas video export, timing, framing, or focus effects.

Camera shots now support optional transition purpose metadata through `outgoingTransitionPurpose`. The field describes the transition from the current camera shot to the next shot in timeline order and supports `orientation`, `reading`, `cinematic`, and `sceneChange`. It is persisted in Project JSON when present and ignored when absent. T0044 does not expose transition purpose UI yet and does not change preview animation, transition timing, focus-region effects, canvas video export, or camera framing.

Project JSON now supports optional `guidedPageOptions` metadata with `showPageEnter` and `showPageExit` booleans. Missing options default to disabled on import for compatibility with older projects. These options affect browser preview playback and canvas video export. Export verification + bug repair added export parity for the existing options without changing Project JSON schema.

Camera shots now support optional `specialEffects` metadata for accepted per-shot special effects. The inspector exposes preset-first controls for None, Shake, Impact Pulse, and Shake + Impact Pulse. Browser preview and canvas export read the active shot's persisted `shake` and `impactPulse` flags directly. Shot Attention Path items now support optional `effectCues` for accepted per-focus-beat Shake and Impact Pulse behavior, with Off, Once, and Repeat controls in the selected-shot inspector. Path items also support optional `effectCueTiming` with Early and Arrival presets for those cues. Browser preview and canvas export apply those cues with the selected timing, defaulting missing timing data to Arrival. Export writes `cameraShots[].specialEffects`, `cameraShots[].attentionPath[].effectCues`, and valid `cameraShots[].attentionPath[].effectCueTiming` only when accepted effect values are enabled. Import accepts only true shot-level `shake` and `impactPulse` flags, only `once` / `repeat` attention cue modes, and only `early` / `arrival` cue timing values; missing, false, malformed, empty, or rejected fields such as `vignette` and `blur` import as effect-off.

The editor canvas now includes a Focus Region drawing mode. When an image and camera shot are selected, users can enable the mode, click-drag on the displayed page to create page-level free-ratio focus regions, select existing focus regions, drag selected focus regions, resize selected focus regions from the bottom-right handle, and delete the selected focus region. Focus region boxes are visually distinct from camera shot boxes. Drawing, dragging, or resizing focus regions does not move or resize camera shots. Loading a new page clears page-level focus regions and resets the default focus-region label/sequence counter so the first new region starts from the initial default pattern again.

The editor canvas now has local view controls for Zoom In, Zoom Out, Pan, and Reset View. Zoom and pan transform the editor stage only and do not mutate stored camera shot or focus region geometry by themselves. Drawing, dragging, and resizing still convert pointer positions back into source image coordinate space.

Camera shots can be dragged within the displayed page. Dragging updates the shot `x` and `y` values in source image coordinate space, keeps width and height unchanged, and clamps the shot inside the source image bounds.

The selected camera shot now shows a bottom-right resize handle. Resizing updates `width` and `height` independently in source image coordinate space, enforces minimum usable dimensions, and keeps the shot inside the source image bounds.

The app now stores camera shots as an array. Users can add multiple shots with the Add Shot control, select one shot at a time, and drag individual shots without moving the others.

The inspector panel now shows the selected shot's ID, label, purpose, purpose-based timing suggestion when assigned, duration, scene hold percentage, focus attention percentage, and read-only source coordinates. Users can edit labels, assign purpose metadata, explicitly apply suggested purpose timing, edit positive duration values, adjust shot-level hold/focus timing balance, and delete the selected shot when more than one shot exists.

The app header now includes an Export JSON action when a page is loaded. The export writes a human-readable schema version 1 JSON file containing image metadata, camera shots in timeline order, and page-level focus regions. The exported image section explicitly marks `binaryIncluded: false` and includes a note that the source image binary is not bundled; the temporary browser object URL is not exported as a durable reference.

The app header also includes an Import JSON action for previously exported project files. Import accepts JSON files, validates schema version 1 and basic shot/focus-region geometry, restores camera shots in imported timeline order, restores page-level focus regions and cinematic fields, and shows a clear message that the source image binary is still separate. If imported image metadata includes dimensions, reselecting a matching image attaches it to the restored project without wiping imported shots or focus regions. If dimensions differ, the app warns and starts a fresh project from the selected image.

When a focus region is selected, the inspector switches to focus-region details. Users can view the focus region ID and source coordinates, edit its label, assign purpose metadata, edit its kind, edit its cinematic effect type, edit its sequence order, and delete the selected focus region without changing camera shot geometry or timing. Focus regions do not expose an apply-timing control because they do not own independent timeline timing yet. Inspector coordinates update after focus region dragging and resizing. Deleting a camera shot does not delete page-level focus regions by default.

The timeline panel shows an ordered list of camera shots in array order. Timeline items show the shot label, ID, and duration, and clicking an item selects the matching shot in the editor and inspector.

Timeline items now include Up and Down controls. These move a shot one position earlier or later in the camera shot array while preserving the selected shot and leaving all shot metadata unchanged.

The app now includes a browser-based preview player with a fixed 16:9 viewport. It shows the uploaded image, uses camera shots in timeline order as flexible subject targets, and provides Play and Stop controls for basic animated playback.

The app header includes explicit Page Enter and Page Exit toggles for browser preview. Page Enter starts playback from a full-page fitted view before moving to the first camera shot. Page Exit moves from the final camera shot back to a full-page fitted view before playback stops. These full-page guide segments do not show focus-region effects and do not mutate shot, focus-region, or transition-purpose data.

Preview playback uses camera shot boxes as subject regions and maps each one into the fixed 16:9 preview with contain-style scaling. The source page image is rendered inside an overflow-hidden shot window whose size is the active shot size multiplied by the contain scale; the image is offset by the active shot origin so only pixels inside the selected shot are visible. During Auto preview focus attention, usable accepted Shot Attention Path items can now apply restrained intra-shot motion inside the active shot window. When a shot has no usable accepted path, preview falls back to the older eligible focus-region behavior. A fallback focus region is eligible for the active camera shot only when at least 60% of the focus region's area intersects that active camera shot region. Preview eligibility is computed from the page-level focus region list, so regions can appear when they mostly fall inside the active shot even if they were originally drawn while another shot was selected.

Preview playback is now phased per camera shot. Each shot uses its `durationMs` as a time budget with travel, scene hold, focus attention, and exit phases. Focus overlays are hidden during travel, hidden during the scene hold, visible only during the focus attention phase, and cleared before the next travel phase. Default timing is 25% travel, 10% scene hold, 60% focus attention, and 5% exit. The first shot has no travel phase; unused travel time is folded into focus attention. Per-shot scene hold and focus attention ratios can now be adjusted in the inspector, with validation that keeps fixed travel/exit behavior safe.

Camera shot travel transitions now interpolate the source-space camera subject center and size before deriving the fixed-stage clipped preview placement. This keeps flexible shot regions, 16:9 stage behavior, and clipped shot-window rendering intact while making movement between varied shot sizes and aspect ratios smoother and more predictable. Travel uses a deterministic smootherstep ease-in/ease-out curve, and focus-region attention transitions remain separate inside the active shot's focus phase.

During the focus attention phase, the active Camera Shot remains the parent reading container and timeline destination. Usable accepted Shot Attention Path items can guide restrained intra-shot pan/push/hold/reveal/emphasis motion inside that parent shot window. Focus Regions still do not replace Camera Shots or become timeline destinations, and shots without usable accepted path items continue to use the prior fallback behavior safely.

During the focus attention phase, eligible focus regions are shown one at a time as enlarged floating cutouts above the main preview image. Each lifted cutout uses actual cropped content from the uploaded image, preserves the focus region aspect ratio, starts from the region's projected position inside the active camera shot, and splits the focus-phase time across eligible regions. Lifted regions are hidden during travel, scene hold, and exit phases.

Focus region effect types now have distinct MVP preview behavior. `lift` is the default and preserves the lifted cutout behavior. Missing effect types fall back to `lift`. `none` skips the cinematic focus effect for that region. `spotlight` dims the active clipped shot window and emphasizes the active focus region without Final mode labels. `zoom` applies a restrained push-in around the active focus region inside the clipped shot-window content, plus a subtle aperture mask that de-emphasizes unrelated shot-window areas without replacing the camera shot with the focus region.

Focus region attention sequencing is controlled by each region's optional `sequenceOrder`. Newly drawn regions receive increasing sequence order values. Preview filters eligible regions using the existing 60% intersection rule, skips `effectType: none`, sorts by sequence order, and falls back to stable page-level array order for regions without explicit sequence values or ties.

Focus region attention now includes simple transition motion between eligible regions inside the focus attention phase. Each active focus-region slice reserves a small entry portion for movement from the previous eligible region. A shared focus-phase dim layer renders whenever an eligible active focus item exists, so background de-emphasis stays continuous while the emphasis geometry changes. Spotlight and zoom interpolate their emphasis geometry from the previous region to the current region when possible. Lift-to-lift handoff keeps the outgoing lift cutout fully visible until the next lift slice begins, then renders outgoing and current lift cutouts as separate keyed elements so the outgoing cutout can fade/drop while the current cutout enters. Regions with `effectType: none` are still skipped before transition sequencing is built.

The preview panel now has a local Final/Debug mode toggle. Final mode is the default and hides debug labels such as focus-region kind text while preserving playback timing, camera shot clipping, contain-fit placement, and focus-region-driven lifted cutouts. Debug mode keeps focus-region kind labels and a more diagnostic focus-lift treatment to help verify eligibility, timing, and projection alignment. The toggle changes preview presentation only and does not mutate project data.

The MVP export readiness plan recommends a browser-side render-specific canvas pipeline with MediaRecorder capture for the first video export prototype. The current DOM/CSS preview should remain the interactive reference and should not be assumed directly recordable. T0032 reuses or mirrors preview timing and placement math, renders frames into a fixed 16:9 canvas, requires the uploaded source image binary to be available in the browser session, and stays scoped to single-page, no-audio, local export.

The app header now includes an Export Video Prototype action when a source image and camera shots are available. The prototype renders a 1280x720 canvas from the current uploaded source image and project state, records the canvas with browser MediaRecorder support, and downloads the best supported browser video format from the available MIME list. MP4 is attempted first when supported; otherwise the prototype falls back to WebM. The export uses camera shots in timeline order, preserves flexible camera shot subject regions through clipped shot-window rendering, mirrors shot duration/timing phases and deterministic camera travel, and includes rough canvas versions of lift, spotlight, and zoom focus-region effects. Focus effect parity is intentionally approximate; the DOM/CSS preview remains the reference for interactive review.

The video export prototype now reports export state while preparing and recording, prevents duplicate export starts while rendering, shows progress and elapsed export timeline time, exposes a Cancel Export action backed by `AbortSignal`, and clearly reports whether the browser is using MP4 recording support or falling back to WebM. Unsupported-browser errors still surface as clear project status errors.

The canvas export renderer now more closely mirrors the DOM/CSS preview for focus-region transitions. Spotlight and zoom export effects use transitioned focus-region geometry, zoom origin/outline placement follows the preview-style transformed focus geometry, shared focus-phase dimming remains continuous across focus slices, and lift-to-lift handoff can render outgoing and current lift cutouts separately. Export parity remains approximate because the canvas renderer still mirrors the preview behavior manually rather than recording the DOM/CSS preview directly.

The video export prototype now has simple header settings for output resolution and FPS. Users can export at 720p or 1080p and choose 24 fps or 30 fps. The default remains 1280x720 at 30 fps, matching the original prototype behavior. These settings only affect the browser-side canvas export and do not change preview playback or Project JSON import/export behavior.

Focus-region cinematic moments now use a stronger, explicit dimming treatment for the surrounding camera-shot background in both DOM/CSS preview and canvas export. Lift, spotlight, and zoom effects share matching dim opacity values between preview and export, with zoom kept slightly lighter to preserve surrounding context. No blur was added, keeping the treatment simple and safer for 1080p export performance.

The app now has an experimental project-level Focus Style control for comparing Clean, Cinematic Dim, and Soft Focus focus-region treatments. The selected style affects both preview and canvas video export by changing background dim strength and focus-region transition pacing, while keeping existing lift, spotlight, zoom, and none effect types intact. This is not a per-region effect editor and is not persisted in Project JSON.

- Preview frame is the fixed 16:9 video stage.
- Camera shot boxes are flexible panel/scene reading containers that preview and future export should contain inside that stage.
- Focus regions are reusable free-ratio page-level attention targets for panels, speech bubbles, faces, action details, or other attention targets.
- Shot Attention Path is the optional ordered per-shot list of references to page-level focus regions for future intra-shot attention/motion.
